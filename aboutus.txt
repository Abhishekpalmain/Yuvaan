2.0 THE PROBLEM: CORPORATE LENDING DATA PARADOX
================================================================================

PROBLEM DEFINITION:
Indian banks face a "Data Paradox": they are overwhelmed by abundant yet
unprocessable data. A single mid-market corporate loan application requires
analyzing:

DATA VOLUME PER APPLICATION:
- Structured Data:
  * Bank Statements: 12-24 months of daily transactions (1000+ rows)
  * GST Filings: GSTR-1, 2A, 2B, 3B (Excel/PDF, complex tables)
  * Returns: Multiple years of financial statements

- Unstructured Data:
  * Annual Reports: 100-300 pages (PDF, scanned images)
  * Board Meeting Minutes: Dense legal text
  * CIBIL Commercial Report: 50-100 pages with complex layouts
  * Legal Notices: Esoteric terminology

- External Intelligence:
  * MCA21 Database: Company master data, director details, charges
  * e-Courts: Pending litigation (civil, criminal, tax)
  * RBI Circulars: Sector-specific regulations
  * News Articles: Negative news screening

CURRENT MANUAL PROCESS:
1. Analyst receives ZIP file with 500+ pages of PDFs and Excel sheets
2. Manually downloads, prints, highlights documents (2-3 weeks)
3. Cross-references GST with bank statements by eyeballing
4. Googles company name for litigation news
5. Writes subjective Credit Appraisal Memo (CAM) in Word
6. Committee reviews (additional week)

PROBLEMS WITH CURRENT PROCESS:
- SLOW: 14-21 days turnaround time
- EXPENSIVE: ₹5,000-10,000 per application in analyst hours
- ERROR-PRONE: Human fatigue causes missed early warning signals
- SUBJECTIVE: No consistent scoring framework
- COMPLIANCE RISK: Black-box decisions fail RBI audits
- FRAUD MAGNET: Sophisticated circular trading and GST fraud goes undetected

COST OF FAILURE:
- Non-Performing Assets (NPAs): ₹100+ Crores annually for mid-sized banks
- Fraud Losses: Billions lost to sophisticated financial statement fraud
- Opportunity Cost: Good customers wait 3 weeks for decision

13. src/ingestor/circular_trading.py
    Purpose: Pillar 1C - Detect circular money transfers (fraud)
    Algorithm (Graph-based):
      Input: bank_statement.json with transactions
      Process:
        G = nx.DiGraph()
        core_entity = company_name
        For each transaction:
          desc = tx['Description'].upper()
          if "TO" in desc:
            target = extract after "TO"
            G.add_edge(core_entity, target, weight=Withdrawal, txn_id=...)
          elif "FROM" in desc:
            source = extract after "FROM"
            G.add_edge(source, core_entity, weight=Deposit, txn_id=...)
        cycles = nx.simple_cycles(G)
        For each cycle in cycles:
          Record cycle.path, cycle.loop_length, transaction details
      Output: Dict{
        company_name, total_entities, total_transactions,
        cycles_detected (bool), cycles: [{path, loop_length, transactions}]
      }
    Returns: Dict[str, Any]
    Complexity: O(n!) worst-case for simple_cycles, but realistic n<100
    Notes: Only tracks edges to/from core company (one-hop relationships)
    Character Count: ~100 lines

15. src/recommendation_engine/generate_cam.py
    Purpose: Pillar 3 - Evaluate, score, explain, generate CAM
    Class: RecommendationEngine
    Methods:
      - __init__(config): Store config, scoring rules, decision thresholds
      - evaluate(gst_results, circular_results, cibil_rank, research_findings) -> evaluation dict
      - _get_recommendation(score) -> (recommendation, limit, rate)
      - _generate_summary(evaluation, explanations) -> str
      - generate_cam_document(evaluation, output_path=None) -> docx_path
    Scoring Algorithm:
      score = base_score (config: 80)
      if gst_results and gst_risk: score += gst_deduction (config: -20)
      if circular_results and cycles_detected: score += circular_deduction (config: -50)
      if cibil_rank > 7: score += poor_cibil_deduction (config: -10)
      if cibil_rank <= 3: score += excellent_cibil_bonus (config: +10)
      if research and litigation_risk in ['High', 'Medium-High']: score += -15 (hardcoded)
      score = clamp(score, 0, 100)
    Decision Logic:
      if score >= 70: APPROVE, limit=INR 5,00,00,000, rate=10.5%
      elif score >= 50: APPROVE WITH CONDITIONS, limit=INR 2,00,00,000, rate=12.0%
      else: REJECT, limit=INR 0, rate=N/A
    CAM Document Structure:
      - Title: "Credit Appraisal Memo (CAM)"
      - 1. Executive Summary (company, score, recommendation, limit, rate, summary)
      - 2. AI Decision Explanation (every adjustment with plain English)
      - 3. The Five Cs of Credit (Character, Capacity, Capital, Collateral, Conditions)
      - 4. Identified Risk Factors (if any)
      - 5. Positive Factors (if any)
      - Footer: Generated by Intelli-Credit
    Formatting: Uses python-docx, styles: 'Intense Quote', 'List Bullet'
    Character Count: ~320 lines

6.0 ALGORITHMS & MATHEMATICAL DETAILS
================================================================================

6.1 GST RECONCILIATION ALGORITHM
=================================

INPUT:
  - GSTR-3B filed return (borrower's claim)
  - GSTR-2A auto-populated (supplier filings)

MATHEMATICAL FORMULATION:
  Let:
    C_IGST = claimed IGST from GSTR_3B
    C_CGST = claimed CGST from GSTR_3B
    C_SGST = claimed SGST from GSTR_3B

    A_IGST = Σ(inv.IGST) for all invoices in GSTR_2A
    A_CGST = Σ(inv.CGST) for all invoices in GSTR_2A
    A_SGST = Σ(inv.SGST) for all invoices in GSTR_2A

  For each tax type t ∈ {IGST, CGST, SGST}:
    if C_t > A_t:
      discrepancy[t] = C_t - A_t
      flag as "OVERCLAIMED"

  Note: Under-claiming (C_t < A_t) is NOT flagged (beneficial to bank)

RATIONALE:
  GSTR-2A is supplier-filed and cannot be manipulated by borrower alone.
  If borrower claims more ITC than suppliers have actually filed, they are
  either:
    - Filing fake invoices (fraud)
    - Making arithmetic errors (incompetence)
  Both signal high risk.

6.2 CIRCULAR TRADING DETECTION ALGORITHM
========================================

INPUT: Bank transactions with descriptions containing "TO" and "FROM"

GRAPH MODELING:
  - Directed graph G = (V, E)
  - Vertices V: Entities (company, vendors, shell companies)
  - Edges E: Financial transactions
    Edge (u, v) exists if money flows from u to v
    Edge attributes: weight=amount, txn_id=identifier

ALGORITHM: nx.simple_cycles(G)
  Returns: List of cycles, where each cycle is a list of nodes forming a loop

DETECTION LOGIC:
  For each cycle:
    If cycle length > 2 OR length == 2 with two edges:
      Consider it suspicious circular trading

  Example cycle: [A, B, C] means:
    A → B → C → A (money circulates and returns to origin)

FRAUD MECHANISM:
  Borrower (A) transfers money to shell company B
  B transfers to shell company C
  C transfers back to A
  Total money in system unchanged but A's bank balance appears higher
  due to the inflow from C, creating artificial impression of revenue.

GRAPH CONSTRUCTION HEURISTIC (current implementation):
  For transaction with description containing "TO X":
    Add edge: company_name → X
  For transaction with description containing "FROM Y":
    Add edge: Y → company_name

  This simplifies to tracking only transactions that directly involve
  the borrower. A full implementation would extract all entity pairs.

COMPLEXITY:
  nx.simple_cycles() uses Johnson's algorithm: O((n + e)(c + 1))
  where n = nodes, e = edges, c = number of cycles
  Realistic: n < 100, e < 1000, so < 1 second

6.3 SCORING MODEL (Explainable AI)
===================================

BASE SCORE: 80 (configurable)

DEDUCTIONS (configurable, defaults):
  GST discrepancy: -20 points
    Trigger: Any CGST/SGST overclaim detected
    Rationale: Tax fraud indicator, serious offense

  Circular trading loop: -50 points
    Trigger: ≥1 cycle detected in bank transaction graph
    Rationale: Direct fraud signal, money laundering pattern

  Poor CIBIL rank (8-10): -10 points
    Trigger: CIBIL Commercial Rank > 7
    Rationale: Historical payment default risk

  High litigation risk (from research): -15 points
    Trigger: Pillar 2 returns "High" or "Medium-High" litigation_risk
    Rationale: Legal exposure, potential liability

ADDITIONS:
  Excellent CIBIL rank (1-3): +10 points
    Rationale: Exemplary credit history

CLAMPING: final_score = max(0, min(100, score))

DECISION THRESHOLDS (configurable):
  APPROVE: score ≥ 70
    Limit: INR 5,00,00,000 (5 Crore)
    Rate: 10.5% p.a. (base rate)

  APPROVE WITH CONDITIONS: 50 ≤ score < 70
    Limit: INR 2,00,00,000 (2 Crore)
    Rate: 12.0% p.a. (risk premium applied)

  REJECT: score < 50
    Limit: INR 0
    Rate: N/A

EXPLANABILITY:
  Every adjustment recorded in evaluation['adjustments'] array:
    {
      "type": "gst_discrepancy" / "circular_trading" / "litigation_risk",
      "value": -20 / -50 / -15,
      "details": {specific discrepancy data}
    }

  CAM document lists each adjustment as bullet points with plain English.

17.0 KNOWN LIMITATIONS & MITIGATIONS
================================================================================

LIMITATION 1: Graph Detection Only One-Hop
Problem: Current circular trading only tracks transactions involving borrower.
          Multi-hop cycles (A→B→C→D→A) where none directly touch borrower missed.
Mitigation: Future: Build complete graph from all extracted entities, not just borrower.

LIMITATION 2: GST Data is Static JSON
Problem: Real GST data comes via API or downloaded Excel, not pre-converted JSON.
Mitigation: Add parsers for Excel (.xlsx) and eventually live GSTN API integration.

LIMITATION 3: OCR Not Implemented
Problem: CIBIL and Annual Reports would require OCR to extract text.
Mitigation: Phase 3: Integrate Tesseract + LayoutLM for Indian documents.

LIMITATION 4: Mock Research (No Real Scraping)
Problem: BeautifulSoup imports exist but data is simulated.
Mitigation: Phase 2-3: Write actual scrapers for MCA, e-Courts (respect robots.txt, rate limits).

LIMITATION 5: No Database
Problem: Decisions not persisted, cannot audit historically.
Mitigation: Phase 2: Add PostgreSQL with decision_history table.

LIMITATION 6: Single-Company Processing
Problem: No batch mode for bulk processing.
Mitigation: Phase 2: Add batch endpoint and queue (Celery/RQ).

LIMITATION 7: No Frontend Upload
Problem: Files must be placed in data/ manually.
Mitigation: Phase 2: Build dashboard.html into full React app with file upload.

LIMITATION 8: Hardcoded CIBIL Rank
Problem: In demo, cibil_rank hardcoded to 4.
Mitigation: Extract from CIBIL PDF using OCR (Phase 3).

LIMITATION 9: Windows Encoding Issues (FIXED)
Problem: Unicode emojis caused console crashes on Windows.
Status: ✅ FIXED - All emojis replaced with ASCII [OK]/[FAIL]/[WARN]
Mitigation: Future: Use colorama for colored output instead of emojis.

LIMITATION 10: No API Authentication
Problem: Anyone can call API (when implemented).
Mitigation: Phase 2: Implement OAuth2/JWT.

CAUSE: Bank statement data has no cycle (or parsing failed)
FIX: Check data/mock_structured/bank_statement.json has cycle:
     TXN9902: "RTGS-OUT TO SHELLCORP CONSULTANTS"
     TXN9903: "RTGS-IN FROM SHELLCORP CONSULTANTS"
     This creates: borrower → shell → borrower cycle

SYMPTOM: "python-docx import error"
CAUSE: Missing dependency
FIX: pip install python-docx

SYMPTOM: "GIT push rejected (non-fast-forward)"
CAUSE: Remote has changes you don't have
FIX: git pull --rebase origin main, resolve conflicts, then push

NBFC: Non-Banking Financial Company

NPA: Non-Performing Asset - loan where borrower hasn't paid for 90+ days
UBO: Ultimate Beneficial Owner - true owner behind shell companies
XAI: Explainable AI - AI that can explain decisions
SHAP: SHapley Additive exPlanations - XAI technique
FastAPI: Modern Python web framework
OCR: Optical Character Recognition
NBFC: Non-Banking Financial Company (lender)
